# Writing Primitive Docs

**Read `STYLE.md` at the repo root before writing.** It is the canonical standard — this skill just frames how to apply it while authoring.

The essentials you're most likely to violate mid-draft:

1. **Know which reader you're serving.** Human docs (`docs/`) teach a developer new to Primitive — concept, first use case, example; don't talk down, don't explain everything. Agent guides (`guides/latest/`) serve an LLM mid-implementation — dense concepts, patterns, examples, reference tables.
2. **Structure pages as a ramp.** Simple at the top (concept → first use → example), advanced and esoteric below. Nothing advanced above anything basic.
3. **Stay at your tier's altitude.** Overview pages name capabilities, never mechanisms. Concept pages show usage, not exhaustive option matrices.
4. **Concepts on their own terms.** One page per concept; never frame feature A as an appendage of feature B; a concept's capabilities live on its own page.
5. **The never-say list** — asides about CLI/console gaps, anything missing or not-yet-supported (including platform-gap callouts — docs show what's there, not what isn't), prior behavior or change history, infra internals, esoteric implementation jargon (no "CRDT" in human docs *or* agent guides — say documents merge concurrent edits automatically; "document store"/"merge semantics" in the guides), unreleased features, guides/ links, private repos, **internal identifiers** (never name a platform-internal type, class, table, or service in reader-facing prose — e.g. `AppGroup`, backend record names; describe concepts in terms of the public client surface only, like `GroupInfo` / `groupId`. A single internal name dropped into an otherwise-fine sentence still leaks — before writing "parallels `X`" or "joined from `X`", confirm `X` exists on the public client API). These arrive one innocuous sentence at a time. Platform differences are shown by parallel examples, never narrated.
6. **Verify every fact** (API names, flags, limits) against the published CLI and `library_repos/js-bao-wss` source; fiction has shipped before. This includes any type name used in prose — confirm it appears on the *public client surface*, not just somewhere in the source.

Mechanics:

- Guides are built artifacts: edit `.template.md` + `examples/` corpus only, then `pnpm render:guides`. Never edit `.ts.md`/`.swift.md`.
- **Compilable client code in a guide template goes in the corpus, included via `{{ example: <subject>/<op> }}` — never as an inline ts/swift fence, and never inside a `{{#lang}}` block.** The corpus is what gives an example its compile gate and its JS/Swift parity enforcement; an inline fence gets neither, and code tucked into a lang block silently exempts itself from both *and* hides from the other language's build. `{{#lang}}` blocks are for language-scoped **prose** (implementation notes, gotchas); inline fences are for what can't compile standalone (shell commands, TOML, output shapes, one-line fragments). If you're writing more than a fragment of ts/swift in a template, stop and add a corpus pair instead. (The renderer copies inline blocks through unchanged — a legacy-migration affordance, not permission for new ones.)
- **Framework glue is generic-first.** A section that integrates a framework (Vue/Pinia, SwiftUI/`PrimitiveApp`) must lead with the compiled, parity-enforced **client** operation (`{{ example: }}` to a normal corpus pair, or a reference to where it's already shown), then present the binding as the wrapper around it. The binding itself can't compile in the corpus harness (it depends on app-template types), so promote it to a corpus file marked `// nocompile`. When the same concept exists in both frameworks (e.g. a data-loader), keep it a parity pair. When it genuinely exists in only one language (a Pinia store has no Swift twin; a `PrimitiveAppState` subclass has no TS twin), also mark the file `// no-parity` and reference it from inside that language's `{{#lang}}` block — `examples.mjs` then skips the cross-language requirement and the renderer keeps it out of the other build. Never invent a fake twin just to satisfy parity. (This is an agent-guide/corpus concern. Human docs aren't built from the corpus; they should emphasize the client-specific code a developer actually writes over a framework's internal wiring, so don't mirror a generic-first guide restructure into them by default.)
- **TOML examples write native TOML — never JSON stuffed into a string.** Operation `definition`/`params` are written as `[operations.definition]` tables and `[[operations.params]]` entries, subscription `params` and workflow object params as tables likewise; `definition = '{"filter":...}'` never appears in an example block in either product. Mention the JSON-string encoding in prose only (recognition + `primitive sync migrate-toml`). Exception: surfaces whose platform contract *is* a JSON-encoded string (prompt test-case fields) — document those as the platform defines them. See STYLE.md's Platform treatment for the full rule.
- Content changed on one side (docs ↔ guides) must be checked on the other.
- Before finishing: `pnpm check:examples`, `npx vitepress build docs`, sidebar/links updated for new or moved pages.
