# Review: `docs/getting-started/blob-buckets.md`

**Tier:** Concept page (human docs). **Verdict:** The page is well-structured and ramps correctly, but it is **factually broken in its two reference tables and its central "CEL Rules" section** — the access-policy enum, the TTL-tier enum, the sync TOML shape, and the signed-URL expiry cap are all wrong, and the entire CEL feature it documents is not a released capability. This is a `--verify` fail: do not ship until the facts below are corrected.

Verification sources: `library_repos/js-bao-wss/src/client/api/blobBucketsApi.ts`, `src/app-api/controllers/blob-buckets-controller.ts`, `src/app-api/controllers/bucket-blobs-controller.ts`, `src/workflows/steps/blob-step.ts`, `cli/src/commands/sync.ts`; published CLI `primitive` **v1.0.49** (`node_modules/.bin/primitive blob-buckets … --help`).

## What is correct (verified clean)

- **CLI Reference (204-227):** every command and flag matches published v1.0.49 — `list`, `get`, `list-blobs`, `upload … --content-type`, `signed-url … --expires`, `delete-blob`, `delete … -y`. The Quick Start CLI `create --key/--name/--access authenticated/--ttl permanent` and `sync push --dir ./config` are all valid in the published CLI.
- **100 MB max object size (231):** matches `ONE_HUNDRED_MB = 100 * 1024 * 1024` (`bucket-blobs-controller.ts:16`, enforced 203/210).
- **Example-corpus includes (50-64, 147-153):** all referenced files exist and the JS/Swift blocks are `corpus-pair` (parity clean per `check-example-parity.mjs`). The JS examples match the client API exactly (`upload`, `getSignedUrl(bucket, blobId, expiresInSeconds)`, `download`, `list({limit})`, `getMetadata`, `delete`). `client.blobBuckets` accessor confirmed (`JsBaoClient.ts:1360`).
- **Workflow steps (157-186):** `blob.upload` (bucketKey/filename/contentType/contentBase64/tags, output includes `blobId`) and `blob.signedUrl` (bucketKey/blobId/expiresInSeconds, output `.url`) match `blob-step.ts`; `email.send` step kind exists (`runner/types.ts:312`).
- **Signed-URL behavioral claims (140-141):** policy re-checked at generation time and bypassed for the recipient during the window — both confirmed (`signedUrl()` calls `checkReadAccess`; `download()` accepts a signed token in lieu of a policy check). Only the "7 days" cap (#6) is wrong.
- **Ramp/altitude & tone:** clean. Opens concept → use cases → Quick Start → policies → TTL → signed URLs → advanced (workflows, vs-document-blobs, CLI, limits). No talking-down, no Cloudflare/R2 internals leaked into prose. No links into `guides/latest/`.

## Bugs to file (gaps found during review — not doc findings)

Check `js-bao-wss` for existing issues first; file there (clients/server/CLI), no assignee/labels unless told.

1. **CEL access control for blob buckets is unenforced.** `BlobBucket` carries `ruleSetId` and the create API/sync accept it, but `checkReadAccess`/`checkWriteAccess` in `bucket-blobs-controller.ts` never evaluate it — access is decided solely by the three-value `accessPolicy` enum. Either wire up rule-set evaluation or remove the dead `ruleSetId` plumbing. (This is the real feature the doc's CEL section was anticipating.)
2. **`owner-only` excludes the uploader.** The policy name implies the uploader can read their own blobs, but only admin/owner role can (`checkReadAccess:109-112`). If uploader-scoped read is intended, `customMetadata.createdBy` is already stored and could gate it. Confirm intended semantics.
